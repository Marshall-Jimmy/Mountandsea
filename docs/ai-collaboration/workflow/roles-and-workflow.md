## 一、AI角色架构

采用「1 + 6 + 2」的AI协作架构：

```
                    总控AI (Orchestrator)
                          |
      ┌───────────┬───────┴───────┬───────────┐
      |           |               |           |
  生存AI       医药AI          修仙AI      祭祀AI
      |           |               |           |
  工匠AI       农牧AI          文化内核AI  平衡AI
                    (横向支撑角色)
```

## 三、协作工作流

### 3.1 日常开发流程

```
1. 总控AI分配任务
       ↓
2. 分支AI领取任务
       ↓
3. 分支AI查询知识图谱（确保设定一致）
       ↓
4. 分支AI生成内容
       ↓
5. 分支AI自查（格式、风格、基本逻辑）
       ↓
6. 提交到各自分支
       ↓
7. 文化内核AI做设定一致性校验
       ↓
8. 平衡AI做数值平衡校验
       ↓
9. 合并到开发分支
```

### 3.2 里程碑流程

```
里程碑开始
    ↓
总控AI制定里程碑计划
    ↓
各分支AI并行开发
    ↓
文化内核AI持续校验设定
    ↓
中期检查（进度+质量）
    ↓
各分支完成开发
    ↓
平衡AI做整体平衡仿真
    ↓
文化内核AI做文化质感抽检
    ↓
总控AI做集成测试
    ↓
里程碑发布
```

## 四、沟通机制

### 4.1 共享知识库
- **知识图谱**：所有设定的唯一真相源
- **设计文档**：各系统的详细设计
- **典籍库**：45部参考典籍

### 4.2 变更通知
- 修改核心设定必须通知文化内核AI
- 影响其他分支的变更必须通知相关AI
- 重大变更由总控AI协调

### 4.3 争议解决
- **设定争议** → 文化内核AI裁定
- **数值争议** → 平衡AI裁定
- **优先级争议** → 总控AI裁定
- 对裁定不服可上诉，由总控AI最终拍板

## 五、代码/数据规范

### 5.1 分支命名
```
feature/[分支名]/[功能名]
示例：feature/survival/basic-gathering
```

### 5.2 提交信息格式
```
[类型]: [简要描述]

类型：
- feat: 新功能
- fix: 修复
- docs: 文档
- data: 数据
- refactor: 重构
```

### 5.3 数据格式
- 所有游戏数据使用JSON格式
- 遵循知识图谱Schema
- 必须有source字段标明出处

## 六、质量保证

### 6.1 每层AI的质量责任
- **分支AI**：对自己产出的内容质量负责
- **文化内核AI**：对设定一致性和文化质感负责
- **平衡AI**：对数值平衡负责
- **总控AI**：对整体进度和集成质量负责

### 6.2 质量检查点
- L1：格式检查（自动化）
- L2：设定一致性检查（文化内核AI）
- L3：数值平衡检查（平衡AI）
- L4：文化质感抽检（文化内核AI+人工）

## 八、协作原则

1. **设定唯一**：知识图谱是唯一真相源，不各自为政
2. **谁开发谁负责**：对自己的产出质量负责
3. **及时沟通**：发现问题尽早说，不要攒到最后
4. **文档先行**：先写设计，再写代码/数据
5. **尊重考据**：所有设定要有典籍依据，不胡编乱造
6. **玩家视角**：永远考虑玩家体验，不要自嗨
