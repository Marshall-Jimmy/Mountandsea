## 一、现状评估与问题诊断

### 1.1 当前仓库内容

| 类别 | 内容 | 状态 |
|------|------|------|
| 核心设定 | 游戏设定集、六大科技树、物品手册、激励方案 | ✅ 丰富 |
| 原文资料 | 山海经全文、45 部古典典籍 | ✅ 完整 |
| 技术调研 | 8 仓库代码级审计报告 | ✅ 完成 |
| 可运行代码 | 无 | ❌ 缺失 |
| 数据表 | 无（物品在 Markdown 里） | ❌ 缺失 |
| 美术资源 | 无 | ❌ 缺失 |

### 1.2 DS 评审核心问题

1. **现在不是游戏，是「企划书」**——有世界观和系统设计，但没有可运行 Demo 来验证手感
2. **范围过大**——一口气想做开放世界 + 程序生成 + 全物种 + 六大科技树 + 修仙 + 祭祀 + 灾变 + 远征，不是小独立游戏规模
3. **技术路线分裂**——设定集写 C++，引擎调研偏 Godot，需尽快统一
4. **物品数据在 Markdown 里**——需要结构化为 JSON 数据表

### 1.3 解决思路

**砍到 MVP，但架构不砍。** JSON 数据表从第一天起设计完整 schema（能承载全部 18 卷），MVP 只填充招摇山子集。系统接口从第一天起定义完整（图鉴 7 槽位、传承完整存档格式），MVP 只启用子集。

## 二、全局架构决策

### 2.1 已确定的技术栈

| 项目 | 决策 | 理由 |
|------|------|------|
| 引擎 | Godot 4.6.1 stable | 最新稳定版，TileMapLayer 成熟 |
| 语言 | GDScript | 引擎调研所有 Godot 仓库均用 GDScript，社区生态好 |
| PCG 框架 | MVP 手写噪声，后期 Gaea addon | 避免前期引入复杂依赖 |
| 数据格式 | JSON（运行时加载为自定义 Resource） | 数据驱动，策划可编辑 |
| 存档格式 | JSON 元数据 + 二进制 chunk 数据 | 人类可读 + 紧凑地形 |

### 2.2 数据驱动设计原则

```
JSON 数据表（data/）  ←→  自定义 Resource 类（scripts/resources/）  ←→  游戏系统（scripts/systems/）
     策划编辑                 代码定义结构/验证                      代码只读，不硬编码
```

- 所有物品/异兽/科技节点/山系/配方数据存储在 JSON 文件中
- 游戏代码通过 DataLoader 单例加载 JSON 并缓存为字典
- 游戏系统只引用 item_id / creature_id，不直接硬编码属性值
- MVP 只填充招摇山数据，但 JSON schema 设计要能承载全部 18 卷内容

### 2.3 渲染路线（取决于视角决策）

| 视角 | MVP 渲染方案 | 长线渲染方案 | 参考项目 |
|------|------------|------------|---------|
| 俯视角 | TileMapLayer + TileSet | 自绘 shader（参考 Substrata） | Gaea |
| 侧视（Terraria 式） | TileMapLayer + TileSet | PackedByteArray → Image → shader → Texture2DArray | Substrata-Game |

**关键设计：Chunk 系统数据层与渲染层解耦。** 无论选择哪种视角，chunk 数据结构相同，只替换渲染层。

## 三、待定事项

> 以下事项需要在进入 Phase 0 之前确定。

### 3.1 视角选择：俯视角 vs 侧视

| 维度 | 俯视角 | 侧视（Terraria 式） |
|------|--------|-------------------|
| 世界模型 | 连续地形，行走自由 | 分层方块，有重力/跳跃 |
| 挖掘编辑 | 不需要（或简化） | 核心玩法（dig/place） |
| 渲染复杂度 | 低（TileMapLayer 足够） | 高（需自绘 shader 支撑大世界） |
| 物理系统 | 简单碰撞 | 重力 + 落体 + 液体模拟 |
| 程序生成 | Gaea 直接可用 | 需参考 Substrata 架构 |
| 与山海经的契合度 | 适合「探索/远征」叙事 | 适合「挖掘/洞穴/地下世界」叙事 |
| MVP 开发速度 | 更快 | 更慢（需自建物理+渲染） |
| 长线扩展性 | 受 TileMapLayer 性能限制 | 自绘 shader 天然支持大世界 |

**建议**：如果优先「最快出原型」→ 俯视角；如果优先「Terraria 式挖掘 + 液体 + 物理」→ 侧视。

### 3.2 美术风格

设定集标注「像素风（讨论中）」。需确定：
- 分辨率：16×16 / 32×32 / 48×48
- 调色板：有限色板（如 PICO-8）还是全彩
- 角色朝向：4 方向还是 8 方向

### 3.3 设定集 C++ → Godot 更新

`docs/山海经游戏设定集.md` 第 16 行「开发语言：C++」需更新为「开发语言：GDScript（Godot 4.6.1）」。

### 5.1 物品数据 `items.json`

```json
{
  "version": "1.0",
  "items": [
    {
      "id": "zhuyu",
      "name": "祝余",

      "effects": [
        {
          "type": "satiate",
          "hunger_restore": 100,
          "duration_hours": 1,
          "condition": "raw"
        }
      ],
      "gather_method": "pick",
      "respawn_time_seconds": 300,
      "stack_size": 99,

        "appearance": "其状如韭而青华",
        "type": "草",
        "effect": "食之不饥",
        "cooking": "熟祝余：食之不饥（时长×4）",
        "location": "招摇山",
        "rarity": "普遍",
        "note": "气味有韭菜味，可致饱腹感"
      }
    }
  ]
}
```

**Schema 要点：**
- `id`：拼音小写下划线，全局唯一
- `type` 枚举：herb / tree / creature_product / mineral / food / equipment / material
- `rarity` 枚举：common / uncommon / rare / precious / legendary
- `effects` 数组：每个效果有 type + 参数 + condition（raw/cooked/equipped）
- `bestiary_slots` 内嵌图鉴 7 槽位数据，避免额外关联表
- `source_text` / `source_book` / `source_mountain` 保留原文出处

## 七、Phase 1：MVP Demo（招摇山 30 分钟可玩）

**目标：** 实现核心循环：随机出生 → 探索招摇山 → 采集/狩猎 → 生火烹饪 → 解锁图鉴 → 遭遇危险 → 死亡/传承。

### 7.1 Chunk 系统（3×3 地图）

- chunk_manager.gd：管理 3×3 chunk（MVP 全部常驻，无加载/卸载）
- chunk_generator.gd：FastNoiseLite 生成招摇山地形（西侧海洋，中央山地/森林）
- chunk_data.gd：单个 chunk 数据结构
- chunk_spawner.gd：在 chunk 内放置采集物和异兽

### 7.2 玩家系统

- player.tscn（CharacterBody2D + CollisionShape2D + Sprite2D）
- player_controller.gd（8 方向移动 + 交互键 E）
- player_stats.gd（hp/hunger/stamina）
- player_inventory.gd（10 格背包）
- player_ui.tscn（HUD：生命条/饥饿条/体力条）

### 7.3 交互与采集系统

- interaction_manager.gd（Area2D 检测 + 交互提示）
- gatherable_component.gd（采集逻辑 + 掉落物品）
- 采集物场景：祝余、迷榖

## 九、Phase 3：完整系统

**目标：** 祭祀、修仙、灾变、远征、完整传承，50+ 小时内容。

### 9.1 祭祀系统

- 各山系祭祀规则（严格遵循原文祠之礼）

### 9.2 修仙系统

- 灵气地绑定机制

### 9.3 灾变系统

- 预兆异兽（长右→洪水，顒→干旱，毕方→野火）

- 数据层不变，只替换渲染层

### Phase 3 验收标准

## 十一、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 视角未确定导致架构返工 | Phase 0 延期 | Chunk 数据层与渲染层解耦，视角决策只影响渲染层 |

| JSON 编辑困难 | 策划难以维护 | 编写验证脚本，后期开发编辑器工具 |
| GDScript 性能瓶颈 | 大世界计算密集 | 关键路径 WorkerThreadPool，后期 C++ 扩展 |
| 美术资源缺失 | 游戏视觉不可接受 | Phase 1 占位资源，同步制作像素素材 |
| 存档格式变更 | 旧存档不兼容 | version 字段 + 迁移器 |
| 范围再次膨胀 | 项目延期 | 严格按 Phase 划分，每个 Phase 有独立验收标准 |
